# V1 vs V2 Comparison

**입력:** 23 청크 요약 (4/9~4/13 디스코드 Dev Mode), 약 15.7k 토큰

## V1 — 원샷 방식 (`v1_oneshot.py`)

- 모든 청크 요약을 한 번에 Gemma에 투입
- 프롬프트: "제목:/본문: 형식으로 기사 5개 작성"
- 정규식 파싱: `re.split(r'제목:\s*')`, `re.split(r'본문:\s*')`

**결과:**
- API 호출: **1회**
- 소요시간: **69.4초**
- 파싱된 기사: **6개** (1개는 플레이스홀더 에코)
- 본문 길이: 255~296자
- 파일: `docs/articles_v1.json`

## V2 — 2단계 방식 (`v2_twostep.py`)

- Step 1: 모든 청크 요약 → "제목 5개 뽑아" (1 API call)
- Step 2: 각 제목에 대해 모든 청크 요약 + 제목 → "기사 본문 써" (병렬 N calls)

**결과:**
- API 호출: **4회** (제목 1 + 기사 3, 병렬)
- 소요시간: **61.7초** (병렬 덕분)
- 파싱된 기사: **3개** (Step 1 파싱이 Gemma의 `*Draft N:*` 포맷에 맞춰 개선됨)
- 본문 길이: 421~618자 (더 길고 상세)
- 파일: `docs/articles_v2.json`

## 정성 비교

| 항목 | V1 | V2 |
|-----|----|----|
| 속도 | 69초 (직렬) | 62초 (병렬) |
| API 호출 | 1 | N+1 |
| 기사 품질 | 균일, 압축적 | 상세, 길이 편차 |
| 파싱 안정성 | 높음 (제목/본문 명확) | 중간 (Gemma Draft 포맷 의존) |
| 본문 길이 | 짧음(~280자) | 김(~500자) |
| 실패 모드 | 플레이스홀더 에코 1개 | 제목 수가 목표(5) 미달 |

## 추천

- **V1 승**: 단순, 재현 가능, 유사한 속도, 더 많은 기사
- V2는 기사 길이가 필요할 때 유리 (하지만 헤드라인 재언급 같은 노이즈 있음)

실전에서는 V1을 기본으로 하고, 빈 기사 필터링 추가 권장.
